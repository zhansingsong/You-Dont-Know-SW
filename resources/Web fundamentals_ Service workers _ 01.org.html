Web fundamentals: Service workers

BY Eero Häkkinen (link is external) ON Nov 28, 2016

Introduction

Traditionally web applications have been able to function only when they are open in a tab within a browser that has a working network connection. This has placed web applications at a disadvantage compared to native applications. Native applications can use locally installed resources to retain partial or even full functionality even offline. Native applications also can use background execution contexts, and can thus also function even when their main user interface view isn’t open. Service workers and related browsers features have recently begun to reduce the disadvantages web applications face by allowing browsers to share some of their native application capabilities with web applications in a controlled manner.

A service worker is a browser concept that provides an event-driven background execution context for a web origin and a scope. As such service workers constitute one of the core components for Progressive Web Apps. A service worker can intercept events related to web pages within its origin and scope. These events include fetch events generated by page navigation and subresource requests (this allows the service worker to act as a programmable proxy), message events generated by the postMessage interface (this allows direct communication between a web page and the service worker which controls the web page), and sync events generated by delayed background synchronization.

Uses cases for service workers include:

  • Resource prefetching
  • Selective and unconditional response caching
  • Fallback responses
  • Mock responses for testing
  • Background data synchronization
  • Load balancing
  • Offline analytics

Service workers are a powerful feature, and therefore there are some security measures in place to prevent them from being abused. First, services workers are available only in secure contexts. Basically pages using service workers must be served via HTTPS.

Second, service workers are origin bound and thus must be served from the same origin as the pages that use them. Therefore, we can’t use content delivery networks (CDNs) for service workers directly. However, service workers can import scripts, and the imported scripts can be from other origins, including CDNs. There are occasionally requests to lift that limitation, but doing so would turn temporary cross-site scripting vulnerabilities into permanent vulnerabilities because service workers can act as proxies.

Finally, the scopes of service workers are limited. By default, a service worker at /app1/sw.js for its origin can only control web pages and intercept subresources under /app1/ but not under /app2/. It is always possible to narrow the scope down whereas conversely, the scope can only be widened if the web server supplies a relevant Service-Worker-Allowed header field.

Slightly simplified service worker lifecycle

Service workers must be registered before they can have any effect. The service worker runs once it’s successfully registered to allow it to add event listeners.

The service worker then reaches an installing state, where the service worker can handle an install lifecycle event. Once the event is handled and all the extend lifetime promises set by the event handler resolve, the service worker becomes installed.

When a previous service worker (if any) is terminated and made redundant, the current service worker reaches an activating state, where the service worker can handle an activate lifecycle event. Once the event is handled and all the extend lifetime promises set by the event handler resolve, the service worker becomes activated. Once activated, the service worker can begin to handle functional events such as fetch events.

A service worker can be terminated at any time when it is not handling an event or if an event handler exceeds the imposed time limits. On the other hand, the service worker will restart if an event arrives while the service worker is terminated. This cycle can happen multiple times per second. This ensures that service workers do not consume too many resources and that the restart cycle is top-of-mind for developers. On the other hand, it also means that service workers do not have a persistent global state, so service workers should utilize data ste, such as IndexedDB, to hold a persistent state instead of using global variables.

Service worker registration

There are three methods you can use to register service workers:

  • Service worker registration using a service worker JavaScript API:
if ('serviceWorker' in navigator) {
    navigator.serviceWorker.register('/app1/sw.js', {scope: '/app1/'})
        .then(function (registration) {
            console.log('Service worker registration succeeded with scope ' + registration.scope);
        })
        .catch(function (error) {
            console.log('Service worker registration failed: ' + error);
        });
}
  • Service worker registration using a link element:
    <link href="/app1/sw.js" rel="serviceworker" scope="/app1/" />
  • Service worker registration using a Link header field in server response:
Link: </app1/sw.js> rel="serviceworker"; scope="/app1/"

The registration promise in the JavaScript API resolves once the service worker is successfully registered. If the service worker is already registered, the promise resolves instantly. That means that it is both safe and preferable to register a service worker on each and every page load.

Service worker installation

A service worker can add a listener for an install lifecycle event.

self.addEventListener('install', event => {
    // ...
});,/pre>

It is safe to use new JavaScript features such as arrow functions in service worker scripts because all browsers that support service workers support those new JavaScript features as well.

The event handler is called while the service worker is at the installing state, during which a previous service worker might still be activated. Therefore, the install lifecycle event handler should only perform nondestructive initialization steps, such as cache population, to avoid disturbing the currently activated service worker.

The initialization steps are often asynchronous, so usually the service worker is not fully initialized when the install lifecycle event handler returns. The event handler can extend the lifetime of the event using the waitUntil method of the event to add a new extend lifetime promises to the event to prevent the service worker from reaching the installed state until all those promises resolve.

By default, the service worker remains at the installed state while a previous service worker is still in use due to having clients. However, it is possible to ignore the existing clients, terminate the previous service worker, and advance activation by calling the skipWaiting method of the service worker while the install lifecycle event is being processed.

Service worker activation

A service worker also can add a listener for an activate lifecycle event.

self.addEventListener('activate', event => {
    // ...
});

The event handler is called while the service worker is at the activating state, during which all previous service workers are already terminated and are made redundant or are about to be made redundant. Therefore, the activate lifecycle event handler can do also destructive initialization steps, such as database schema updates and cache purges.

Once again, to prevent the service worker from reaching the activated state too early, the event handler can extend the lifetime of the event using the waitUntil method of the event to add a new promise to the extend lifetime promises of event and hence delay the activated state until all those promises resolve.

The activated service worker does not control all existing clients and receive functional events by default, but only after navigation or reload. However, it is possible to force the activated service worker to take control of all existing clients instantly by calling the clients.claim method of the service worker while the activate lifecycle event is being processed.

Functional and message events

Once a service worker is activated, it is ready to receive function events (such as fetch, foreignfetch, and sync events) and message events. The fetch and foreignfetch events are generated by page navigation and subresource requests and allow the service worker to act as a programmable proxy. The message events are generated by the postMessage interface and allow direct communication between a web page and its controlling service worker.

Only the fetch, foreignfetch, and message events are defined in the service worker specification. However, as functional events provide a useful framework to build on, other specifications define other functional events. For instance, the background synchronization specification defines the sync events which are used for delayed background synchronization.

A service worker as a programmable proxy

When a service worker is used as programmable proxy, the main duty of the install lifecycle event handler is to populate a cache with core resources.

const CACHE_VERSION = 'v2';
self.addEventListener('install', event => {
    // Prefetching.
    event.waitUntil(
        // Open a cache for resources.
        caches.open(CACHE_VERSION)
            .then(cache => {
                // Begin fetching the resources.
                // The returned promise resolve only when all
                // of them are fetched and cached.
                return cache.addAll([
                    '/app1/index.html',
                    '/app1/resources/v2/background.png',
                    '/app1/resources/v2/fallback.png',
                    '/app1/resources/v2/script.js',
                    '/app1/resources/v2/style.css'
                ]);
            })
            .then(() => {
                // Optional. Now that all previous tasks are
                // done, advance to activation immediately.
                return self.skipWaiting();
            });
    );
});

The main duty of the activate lifecycle event handler is to purge old cached resources that are not being used any more. It is important to do this because otherwise at some point the browser will determine that the origin is using too much cache resources and will purge the entire cache. By purging the cache early, the service worker can do selective purging.

self.addEventListener('activate', event => {
    event.waitUntil(
        // Purge old cache versions.
        caches.keys()
            .then(keys => Promise.all(keys.map(key => {
                if (key != CACHE_VERSION)
                    return caches.delete(key);
            })))
            .then(() => {
                // Optional. Now that all previous tasks are
                // done, take control of all existing clients.
                return self.clients.claim();
            });
    );
});

The fetch event handler does the proxying. The example code does unconditional caching and falls back to fetching from a server first and then to a fallback response.

self.addEventListener('fetch', event => {
    // Unconditional caching.
    event.respondWith(
        caches.match(event.request).then(function(response) {
            return response || fetch(event.request)
                .then(response => {
                    // Open a cache for resources.
                    caches.open(CACHE_VERSION).then(cache => {
                        // Cache the response.
                        cache.put(event.request, response);
                    });
                    // Return the response.
                    return response.clone();
                })
                .catch(() => {
                    // Return a fallback response.
                    return caches.match('/app1/resources/v2/fallback.png');
                });
        );
    );
});

This kind of prefetching and unconditional caching can be implemented using a cache manifest. However, the power of fetch events lies with programmability. The event handler can examine the properties of requests (such as the URL and the request and response header fields) and act differently in the case of different request types. For example, the service worker could prefer cached responses and fall back to fetched ones for some requests, prefer fetched responses and fall back to cached ones for some other requests, and use load balancing for yet other requests.

Issues with proxying service workers and possible solutions

Service workers may utilize local caches, which can make repeated resource loadings much faster. However, the same does not usually hold when caches are cold during initially load resources. There are actually two issues here.

One issue is that because code evaluation takes time, programmability inevitably causes a delay. One solution flying around to solve this problem is using static routes, which allows service workers to declare that certain resource requests are always handled in the same simple way. It might be possible to cover the most common cases with just a few chained models. This would allow service workers to use declarative routes for most resources and to use fetch event handlers only for the most complex resource requests. However, because this issue hasn’t been extensively studied and we believe this issue is usually fairly minor, this solution has not been worked on.

The other issue is caused by the fact that fetch event queue handling is not parallelized although (properly) implemented fetch event handlers are asynchronous. A typical web browser uses multiple connections and threads to fetch network resources. However, with service workers, a single service worker acts as the gatekeeper for these fetches and can become a bottleneck. The solution we’re discussing to solve this problem, is to have fetch events be handled in multiple threads in parallel. This discussion is ongoing, but we will likely implement this solution in the near future.

Summary

Service workers have bought the set of execution contexts available to web applications closer to those of native applications. Service workers have allowed web applications to locally run part of their functionality that has traditionally been able to run only on remote servers. This has narrowed the feature and performance gap between web and native applications. However, more complementary standards are needed before web and native applications become indistinguishable.

Service workers are essential for Progressive Web Apps. Service workers provide connectivity independence by allowing apps to work offline and to handle errors caused by low quality networks. They ensure freshness of apps by allowing apps to use service worker update process and they contribute to discoverability of apps along Web App Manifest, too.
